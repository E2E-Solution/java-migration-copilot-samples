Here is a complete, step-by-step guide to delivering a compelling live demo of the **Asset Manager** workshop using the GitHub Copilot App Modernization for Java extension.

This guide is structured to help you show the audience the end-to-end journey: assessing a legacy app, migrating its components, and validating the changes using AI.

`Can you help me analyze the #codebase and #file:asset-manager for showcasing Java App modernization`

### Phase 1: Pre-Demo Setup (Do this before you present)

To ensure a smooth live demo, you want your environment ready to go without making the audience watch progress bars.

1. **Install Prerequisites:** Ensure you have VS Code (v1.106+), Java 21+, Maven/Gradle, and an active GitHub Copilot subscription.
2. **Install the Extension:** Install the **GitHub Copilot app modernization** extension from the VS Code marketplace and restart your IDE.
3. **Clone the Repository:** Run `git clone https://github.com/Azure-Samples/java-migration-copilot-samples.git` in your terminal.
4. **Open the Project:** Open the `asset-manager` folder directly in VS Code.
5. **Test the Build:** Run a quick local `mvn clean install` just to ensure your local Java environment is fetching dependencies correctly before you get on stage.

---

### Phase 2: The Live Demo Script

#### Step 1: Set the Stage for the Audience

Start by opening the `asset-manager` codebase and explaining the scenario to your audience.

* **Explain the App:** "This is the Asset Manager, a legacy Spring Boot application. Right now, it stores user-uploaded images using **Amazon S3**, handles messaging with **RabbitMQ**, and writes to a local database."
* **State the Goal:** "Our goal today is to modernize this app for the cloud. We want to migrate S3 to **Azure Blob Storage**, RabbitMQ to **Azure Service Bus**, and the database to **Azure Database for PostgreSQL**—all using GitHub Copilot."

#### Step 2: Run the Cloud Readiness Assessment (AppCAT)

Show the audience how Copilot acts as an architect to plan the migration.

1. Open the **GitHub Copilot app modernization** pane on the VS Code Activity Bar.
2. Click **Run Assessment** under the ASSESSMENT section.
3. Explain that Copilot is now running AppCAT (Application and Code Assessment Tool) in the background.
4. Click **Continue** in the chat prompt when it asks to verify the environment and run the logs.
5. **Show the Result:** Open the generated Assessment Report. Highlight the detected cloud-readiness issues and the recommended predefined tasks.

#### Step 3: Execute a Predefined Migration Task

Pick one specific, highly visual migration to show the tool in action. Moving from RabbitMQ to Azure Service Bus is a great choice.

1. In the Assessment Report, locate the task for migrating messaging and click **Run Task**.
2. Show the audience how the Copilot chat window switches to **Agent Mode**.
3. Point out the generation of the `plan.md` and `progress.md` files. Explain: "Copilot isn't just generating code snippets; it's creating a step-by-step architectural plan before it touches our code."
4. Type **continue** in the chat box to authorize the agent to start.
5. **Highlight the Git Workflow:** Point out that Copilot automatically checks out a new branch for the migration, keeping the main branch safe.

#### Step 4: Show the AI Validation Loop

This is the most powerful part of the demo because it shows Copilot acting like a real developer. As the agent runs, narrate the five-part loop happening on screen:

* **Code Transformation:** Show how Copilot swaps out the old RabbitMQ dependencies in the `pom.xml` for Azure Service Bus and updates the Java config files.
* **Build-Project Tool:** Explain that Copilot is compiling the code to catch errors.
* **Run-Test Tool:** Highlight that Copilot is running the unit tests and will automatically fix any tests that break due to the new architecture.
* **Validate-CVEs Tool:** Point out that Copilot is scanning the new dependencies for vulnerabilities to ensure the modernized app is secure.

#### Step 5: Review the Final Summary

Wrap up the technical execution by showing the final results.

1. Open the `summary.md` file generated by the agent.
2. Walk the audience through the summary of changes, highlighting how many files were updated and the tests that passed.
3. Open the VS Code Source Control tab to show the clean diff of the codebase, proving that the migration was precise and isolated to the correct files.

### Phase 0: Baseline and Assessment

Goal: establish a baseline and generate an actionable plan.

1. Run AppCAT assessment from the extension.
2. Capture the recommended tasks and confirm service mapping above.
3. Validate the current baseline build with Maven.

### Phase 1: Runtime and Framework Upgrade

Goal: unlock Azure SDKs and Spring Boot 3.x support.

1. Upgrade Java 8 to Java 21 in the parent build configuration.
2. Upgrade Spring Boot 2.7 to 3.x and migrate javax to jakarta.
3. Confirm build success and resolve any deprecations or API changes.

Key areas affected: [asset-manager/pom.xml](asset-manager/pom.xml), [asset-manager/web/pom.xml](asset-manager/web/pom.xml), [asset-manager/worker/pom.xml](asset-manager/worker/pom.xml), [asset-manager/web/src/main/java/com/microsoft/migration/assets/config/WebMvcConfig.java](asset-manager/web/src/main/java/com/microsoft/migration/assets/config/WebMvcConfig.java)

### Phase 2: Data Layer Migration (PostgreSQL)

Goal: move metadata storage to Azure Database for PostgreSQL Flexible Server.

1. Replace local JDBC settings with Azure Postgres connection string and SSL requirements.
2. Move secrets to Key Vault (or env vars for demo) and enable Managed Identity access.
3. Validate schema creation and data flow on upload/delete events.

Key areas affected: [asset-manager/web/src/main/resources/application.properties](asset-manager/web/src/main/resources/application.properties), [asset-manager/worker/src/main/resources/application.properties](asset-manager/worker/src/main/resources/application.properties), [asset-manager/web/src/main/java/com/microsoft/migration/assets/model/ImageMetadata.java](asset-manager/web/src/main/java/com/microsoft/migration/assets/model/ImageMetadata.java)

### Phase 3: Storage Migration (S3 to Blob Storage)

Goal: replace AWS S3 SDK with Azure Storage SDK and MSI.

1. Replace AWS SDK dependencies with Azure Blob Storage dependencies.
2. Replace `AwsS3Service` and `S3FileProcessingService` with Blob Storage implementations.
3. Migrate storage configuration to Azure and remove access key usage.
4. Validate list, upload, view, delete, and thumbnail paths.

Key areas affected: [asset-manager/web/src/main/java/com/microsoft/migration/assets/service/AwsS3Service.java](asset-manager/web/src/main/java/com/microsoft/migration/assets/service/AwsS3Service.java), [asset-manager/worker/src/main/java/com/microsoft/migration/assets/worker/service/S3FileProcessingService.java](asset-manager/worker/src/main/java/com/microsoft/migration/assets/worker/service/S3FileProcessingService.java), [asset-manager/web/src/main/resources/application.properties](asset-manager/web/src/main/resources/application.properties)

### Phase 4: Messaging Migration (RabbitMQ to Azure Service Bus)

Goal: move queue-based workflow to Service Bus and formalize retry behavior.

1. Replace Spring AMQP dependencies with Azure Service Bus client and Spring integration.
2. Replace Rabbit config with Service Bus queue configuration.
3. Implement retry and dead-letter handling using scheduled delivery or DLQ rules.
4. Validate the worker’s processing acknowledgements and error routing.

Key areas affected: [asset-manager/web/src/main/java/com/microsoft/migration/assets/config/RabbitConfig.java](asset-manager/web/src/main/java/com/microsoft/migration/assets/config/RabbitConfig.java), [asset-manager/worker/src/main/java/com/microsoft/migration/assets/worker/config/RabbitConfig.java](asset-manager/worker/src/main/java/com/microsoft/migration/assets/worker/config/RabbitConfig.java), [asset-manager/worker/src/main/java/com/microsoft/migration/assets/worker/service/AbstractFileProcessingService.java](asset-manager/worker/src/main/java/com/microsoft/migration/assets/worker/service/AbstractFileProcessingService.java)

### Phase 5: Identity and Secrets

Goal: eliminate static secrets and align to Azure identity.

1. Enable Managed Identity for the app runtime.
2. Store connection strings and keys in Key Vault and reference via environment/App Configuration.
3. Remove plaintext secrets from properties.

Key areas affected: [asset-manager/web/src/main/resources/application.properties](asset-manager/web/src/main/resources/application.properties), [asset-manager/worker/src/main/resources/application.properties](asset-manager/worker/src/main/resources/application.properties)

### Phase 6: Observability and Health

Goal: production readiness for containers and cloud diagnostics.

1. Add Spring Boot Actuator and expose health endpoints.
2. Connect logs and metrics to Azure Monitor and Application Insights.
3. Confirm readiness/liveness probes for the web and worker services.

Key areas affected: [asset-manager/web/pom.xml](asset-manager/web/pom.xml), [asset-manager/worker/pom.xml](asset-manager/worker/pom.xml)

### Phase 7: Containerization and Deployment

Goal: build, containerize, and deploy to Azure.

1. Create Dockerfiles for web and worker modules.
2. Deploy to Azure Container Apps (default) or AKS.
3. Wire Service Bus, Blob, Postgres, and Key Vault to runtime settings.

Key areas affected: [asset-manager/pom.xml](asset-manager/pom.xml)

### Phase 8: Validation and Hardening

Goal: confirm stability and compliance after migration.

1. Run build and tests; fix regressions.
2. Run CVE scans and update dependencies as needed.
3. Validate data integrity, message processing, and thumbnail generation.

Key areas affected: [asset-manager/worker/src/test/java/com/microsoft/migration/assets/worker/service/S3FileProcessingServiceTest.java](asset-manager/worker/src/test/java/com/microsoft/migration/assets/worker/service/S3FileProcessingServiceTest.java)
